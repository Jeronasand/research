  • 端点数量：有文档写 83 endpoints，MCP 页写 86 endpoints，Agent 入口又出现 90+ commands。这通常意味着 OpenAPI 持续在更新，而文案没有完全同步。

一些很有价值的细节信号

  • Polymarket-Kalshi 匹配表 的说明中写到：跨平台匹配由 LLM matching (Grok) 生成并带 confidence scoring。这意味着 AskSurf 的某些“统一视图”本身就使用了 LLM 做数据对齐，而不只是纯规则 ETL。
  • Kalshi 数据 被官方明确标注为 centralized exchange data，不能 on-chain 验证；而 Polymarket 则有明确的 Polygon / on-chain event 语义。这一点对你做“可追溯性/可信度分层”很重要。
  • 价格表 明确写了 CoinGecko 为稳定参考源，而 DEX VWAP 有 wash trading / 流动性偏斜风险。这意味着 AskSurf 自己也承认“价格不是一条真值流”，而是按优先级合成。

调用规范与最佳实践

通用规范

认证
统一使用 Authorization: Bearer <API_KEY>；不要硬编码到代码仓库里，官方多处明确建议从环境变量读取。
分页
所有 list endpoints 通常支持 limit 与 offset，默认 limit=20，最大 100。
链名
必须用 canonical long-form：ethereum、solana、base、arbitrum 等；不要用 eth / sol / matic。
symbol
大写、逗号分隔：BTC、ETH,SOL。
时间范围
常见是 7d / 30d / 365d / max。
错误码
普遍可见 400 / 401 / 404 / 422 / 429 / 502；Chat 端点页还出现 402，但语义未公开详细解释。
计费
采用 credits；多数响应会带 meta.credits_used；余额查看有 GET /v1/me/credit-balance 的文档线索。

Chat API 最佳实践

  • 如果需要 agent-style 行为，优先使用 surf-1.5 / surf-1.5-thinking；若追求速度，用 surf-1.5-instant。
  • surf-research 与 surf-1.5 官方建议把 timeout 设到 600 秒。
  • 若要控制模型能力边界，可使用 ability；若要输出引用风格，可使用 citation。
  • 如果你已经有 OpenAI SDK 代码，Surf 的接入成本会很低，因为它走 OpenAI-compatible request shape。

REST Data API 最佳实践

  • 以“typed endpoint + 标准分页 + 标准 error shape”的方式接入，优于直接从自然语言结果里萃取 JSON。
  • 多链钱包类接口建议先从 wallet/detail 做聚合视图，再按需下钻 history / transfers / protocols。
  • Web 搜索接口已经支持域名过滤 site；如果你做投研系统，应该从第一天起就把新闻域名、官方域名白名单化。

Onchain SQL / Data Catalog 最佳实践

  • 官方明确建议：先 filter block_date，因为它通常是分区键，是 ClickHouse 性能最关键的杠杆。
  • 尽量避免 SELECT *；大表上显式列投影更稳。
  • ReplacingMergeTree 表需要 FINAL；并且文档特意提醒 FINAL 的位置写法。
  • 做跨表分析时，各表先在 CTE 中独立过滤，再 JOIN，会比“先 JOIN 后过滤”更合理。

CLI / Skill / MCP 最佳实践

  • CLI：首次会话先 surf sync，后用 surf list-operations 和 surf <command> --help 自证参数。
  • CLI 用在脚本里时，始终加 -o json -f body.data，避免解析人类可读输出。
  • Skill：本质是 instruction layer，不会替你发明新数据能力；它最大的价值是减少 agent 乱用参数。
  • MCP：适合多工具代理统一编排；但因为它受 OpenAPI 漂移影响，最好把“tool signature 变更检测”纳入你的集成测试。

Chat API

步骤：获取 API Key → 设置环境变量 → 选模型 → 发 JSON → 处理 JSON 或 SSE → 给深度模型加长超时。

async function askSurfChat() {
  const resp = await fetch("https://api.asksurf.ai/gateway/v1/chat/completions", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${process.env.SURF_API_KEY}`,
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      model: "surf-1.5",
      messages: [
        { role: "user", content: "总结今天 BTC 市场状态，并列出 3 个主要驱动因素" }
      ],
      reasoning_effort: "medium",
      stream: false,
      ability: ["market_analysis"],
      citation: ["source"]
    })
  });

  if (!resp.ok) {
    throw new Error(`HTTP ${resp.status}: ${await resp.text()}`);
  }

  const data = await resp.json();
  console.log(data.choices?.[0]?.message?.content);
}

如果你是直接改 OpenAI SDK，官方示例的思路是：baseURL = "https://api.asksurf.ai"，然后照 OpenAI Chat Completions 方式调用；但生产前请核对当前 Chat 路径是否仍维持该兼容映射。

Onchain SQL

curl --request POST \
  --url https://api.asksurf.ai/gateway/v1/onchain/sql \
  --header "Authorization: Bearer $SURF_API_KEY" \
  --header "Content-Type: application/json" \
  --data '{
    "sql":"SELECT project, sum(amount_usd) AS volume_usd FROM agent.ethereum_dex_trades WHERE block_date = today() - 1 GROUP BY project ORDER BY volume_usd DESC LIMIT 10",
    "max_rows":1000
  }'
async function runOnchainSql(sql) {
  const resp = await fetch("https://api.asksurf.ai/gateway/v1/onchain/sql", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${process.env.SURF_API_KEY}`,
      "Content-Type": "application/json"
    },
    body: JSON.stringify({ sql, max_rows: 1000 })
  });
  if (!resp.ok) throw new Error(await resp.text());
  return resp.json();
}

最终判断
如果你的目标是“围绕 AskSurf 做二次开发/数据接入”，最可靠的方案是：先对 REST Data API 和 Onchain SQL 做 PoC，再决定是否上 Chat API 做自然语言层，以及是否用 MCP/Skill 把它接进 agent 工作流。不要一开始就只盯着首页文案；真正的产品能力边界，绝大多数都写在 docs.asksurf.ai。
