推荐集成架构

针对你现在的方向，建议把 Dune 放在“链上数据层”和“AI Agent Tool 层”，通过本地 Go 服务做权限、缓存、成本和通知控制。

架构 A：Codex / Agent 直接使用 MCP

Codex / Claude / Cursor
  ↓ MCP
Dune MCP Server
  ↓
Dune Data Warehouse / Query Engine
  ↓
结果返回给 Agent

优点：接入快，适合研究和探索。缺点：成本和权限需要严格控制。

架构 B：本地 Go 服务代理 Dune API

Codex / 前端 / AI Pulse
  ↓
Local Go API
  ↓ 权限控制 / Query 白名单 / 缓存 / 成本限制
Dune Data API
  ↓
Dune Query / MV / Pipeline
  ↓
Local DB / Notification / Dashboard

这是更适合生产的模式。API Key 只放在本地服务中，不暴露给 Agent 或浏览器。

架构 C：Dune Webhook → AI Pulse

Dune Scheduled Query
  ↓ Webhook
ai-pulse-server
  ↓ 规则判断
Flutter App / 手机通知 / Slack / 飞书

适合把“链上指标变化”变成事件通知。

本地 Docker 建议

services:
  ai-pulse-dune:
    image: ai-pulse-dune:latest
    environment:
      - DUNE_API_KEY=${DUNE_API_KEY}
      - MAX_QUERY_DAYS=7
      - MAX_ROWS=1000
      - WEBHOOK_SECRET=${WEBHOOK_SECRET}
    ports:
      - "8080:8080"

接口设计建议

本地接口说明
POST /api/dune/query/run执行白名单 Query。
GET /api/dune/query/:id/latest获取 Query 最新结果。
POST /api/dune/webhook/:secret接收 Dune Webhook。
GET /api/dune/usage查看 credits 用量。
POST /api/dune/signal/evaluate把结果转成 AI Pulse 事件。
